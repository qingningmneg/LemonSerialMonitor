# Lemon串口监控设计说明

- 日期：2026-07-10
- 状态：用户已确认总体设计
- 目标平台：Windows 10/11 x64
- 使用方式：个人自用，测试签名驱动

## 1. 项目目标

实现一套免费、无需独占串口的 Windows 串口监控分析软件。软件以 CEIWEI CommMonitor 公开用户手册描述的功能为参考，独立实现其核心能力，并把快速、灵活的数据复制作为重点体验。

最终版本需要具备：

1. 在其他程序已经打开 COM 口时捕获其串口通信，不改变原程序的串口配置。
2. 同时监控多个标准串口、USB 转串口、蓝牙串口和内核虚拟串口。
3. 捕获 Read、Write、主要 IRP 和全部标准串口 IOCTL 请求及其参数。
4. 提供列表、Line、Dump、终端和 Modbus 五种联动视图。
5. 支持会话保存与恢复、过滤、搜索、自动重定向和多格式导出。
6. 支持 ANSI、UTF-7、UTF-8、UTF-16LE/BE 等编码，以及 2、8、10、16 进制显示。
7. 支持 Modbus RTU/ASCII 帧重组、解析、请求响应匹配和校验。
8. 所有视图都能方便地复制单元格、数据包、选中行和全部记录。
9. 提供完整的安装、卸载、使用和故障排查说明。

参考资料：

- [CommMonitor 公开功能清单](https://www.ceiwei.com/cn/help/31)
- [CommMonitor 用户手册目录](https://www.ceiwei.com/mt/cn/help/32)
- [CommMonitor 各版本功能比较](https://www.ceiwei.com/cn/help/30)

本项目只参考公开的功能描述，不复制参考软件的源代码、二进制、图标、商标、界面素材或私有会话文件格式。

“完整功能”以参考软件独立桌面版公开用户手册为对齐基线，不包含单独销售的 SDK/OCX 所提供的数据阻断、修改或主动注入接口。

## 2. 范围和限制

### 2.1 包含范围

- Windows COM 设备栈中的串口通信监控。
- 对真实物理 COM 口、USB 转串口、蓝牙串口和虚拟 COM 口的支持。
- 监控程序、后台服务、内核过滤驱动、安装脚本和用户文档。
- 只读监控。首轮完整交付不提供修改、阻断或注入串口数据的能力。

### 2.2 不包含范围

- Windows XP/7/8、Windows Server、ARM64 和 32 位系统。
- 绕过安全启动、代码完整性保护或其他 Windows 安全机制。
- 非 COM 接口的厂商专用 USB 协议监控。
- 对第三方私有协议进行未授权逆向分析。
- 生产级微软签名。个人使用版本采用测试证书和 Windows 测试模式。

### 2.3 环境约束

- 开发和运行目标为 Windows 10/11 x64。
- 初次安装需要管理员权限。
- 测试签名模式通常要求关闭 UEFI 安全启动并重启电脑。
- 安装或卸载串口类过滤驱动后可能需要重启串口设备或电脑。

## 3. 方案选择

### 3.1 选定方案：KMDF 串口上层过滤驱动

使用 KMDF 编写串口设备类上层过滤驱动。驱动透明转发所有请求，在不改变请求语义的前提下复制监控数据。该方案能观察已经被其他程序独占打开的串口，并能覆盖普通应用层 Hook 无法观察的驱动访问。

### 3.2 未选方案

- 旧式 WDM 过滤驱动：控制能力强，但代码复杂，内存和并发错误更容易导致系统崩溃。
- 用户态 API Hook：可以快速做出演示，但只能观察被注入的特定进程，容易漏掉直接系统调用、服务和驱动发出的请求。
- 虚拟串口代理：稳定且易开发，但要求修改原程序的端口配置，不符合“不占用、不改连接”的核心目标。

## 4. 总体架构

系统划分为五个边界清晰的组件。

### 4.1 `CommMonitorFilter.sys`

KMDF 串口上层过滤驱动，职责如下：

- 附加到 Windows Ports 类设备栈，透明转发 PnP、电源和串口 I/O 请求。
- 在 Write 请求下发前复制发送缓冲区，并在完成后记录实际完成长度和状态。
- 为 Read 请求注册完成处理，在成功完成后复制实际接收字节。
- 捕获 Create、Close、Cleanup、Read、Write、Flush、DeviceControl 等主要 IRP。
- 捕获标准串口 IOCTL 的输入、输出、完成状态和关键配置值。
- 记录单调递增序号、高精度时间、PID、端口标识、请求类型、状态和数据。
- 根据服务下发的端口、IRP 和 IOCTL 位图决定是否生成监控事件。
- 把事件写入固定容量的非分页环形缓冲区。

驱动不得等待用户态程序，不得把监控错误返回给原请求。环形缓冲区满时只丢弃监控事件，并累加丢包计数。

### 4.2 `CommMonitor.Service`

.NET 8 Windows 后台服务，作为驱动和桌面程序之间的唯一桥梁：

- 打开驱动控制设备并批量读取事件。
- 管理开始、暂停、停止以及监控端口和内核过滤条件。
- 补充进程名称、COM 口友好名称等用户态信息。
- 对事件进行版本校验、排序、统计和批量持久化。
- 通过受限命名管道向桌面程序提供命令和事件流。
- 桌面程序退出后仍能按用户设置继续保存捕获会话。

驱动控制设备只允许 LocalSystem 和管理员访问；普通桌面程序通过服务提供的命名管道访问，不直接控制驱动。

### 4.3 `CommMonitor.Core`

不依赖 UI 的 .NET 8 类库：

- 定义统一的 `CaptureEvent`、会话、过滤、显示和导出模型。
- 实现字节格式化、文本解码、搜索、合并和复制格式化。
- 实现 Modbus RTU/ASCII 帧重组、校验和字段解析。
- 实现会话存储、导入、导出和实时重定向。
- 允许使用纯单元测试验证绝大多数业务逻辑。

### 4.4 `CommMonitor.App`

.NET 8 WPF x64 桌面程序，采用 MVVM：

- 多会话标签页，每个会话可选择一个或多个 COM 口。
- 工具栏提供新建、打开、保存、开始、暂停、停止、过滤、搜索和复制。
- 使用虚拟化列表和增量加载，避免大量记录导致界面冻结。
- 所有视图共享同一个事件选择模型，实现数据包联动定位。
- 用户设置保存在当前用户目录，不要求管理员权限。

### 4.5 安装和文档

- 开发阶段提供管理员 PowerShell 安装、卸载、状态检查和恢复脚本。
- 完整交付阶段使用免费安装工具制作安装包，负责程序、服务和驱动包的安装与卸载。
- 测试模式和安全启动的变更必须由用户明确执行，安装程序只检测并给出说明，不偷偷修改固件安全设置。
- 文档包括快速开始、完整安装、卸载恢复、用户手册、常见问题和开发构建说明。

## 5. 事件模型和数据流

每条捕获记录至少包含：

- 事件序号和会话序号。
- 内核高精度时间戳、换算后的本地时间和捕获来源。
- PID、进程名称、COM 口和设备实例标识。
- IRP 主功能、IOCTL 名称和数值、方向、请求状态。
- 请求长度、实际完成长度和原始数据字节。
- 串口配置快照，包括波特率、数据位、停止位、校验位和流控状态。
- 丢包、截断、解码错误等标志。

数据流如下：

1. 原程序向串口设备栈提交请求。
2. 过滤驱动复制监控所需信息并立即把原请求继续下发。
3. Read 和需要结果的 IOCTL 在完成处理阶段补充实际数据与状态。
4. 驱动把事件写入环形缓冲区，服务批量读取。
5. 服务补充用户态元数据并批量写入会话数据库。
6. 桌面程序通过命名管道订阅新增事件，更新虚拟化视图。
7. 视图、复制、搜索和导出只读取统一事件模型，不反向影响驱动。

## 6. 会话存储

会话使用独立的 SQLite 文件，建议扩展名为 `.cmsession`：

- 使用 WAL 和批量事务降低持续写入开销。
- 原始数据以 BLOB 保存，不因当前显示编码而丢失信息。
- 为事件序号、时间、端口、PID 和请求类型建立索引。
- 元数据保存应用版本、事件格式版本、视图设置和过滤条件。
- 打开旧版本会话时只读兼容；需要迁移时先生成备份。

导出格式包括 TXT、CSV、HTML、JSON 和原始二进制。实时重定向支持文本、CSV、JSON Lines 和原始二进制，并采用临时文件加安全替换，降低异常退出造成的损坏。

## 7. 监控控制语义

- 开始：启用驱动事件生成，序号从 1 开始或从恢复会话的末尾继续。
- 暂停：驱动暂时不生成事件，已显示和已保存数据保持不变。
- 继续：恢复生成事件，序号继续递增。
- 停止：停止生成事件并结束当前捕获段，不自动删除已有记录。
- 清空：单独命令，明确确认后才删除当前会话数据并将显示序号复位。

“停止”和“清空”分开，避免参考软件停止时清缓存导致误丢数据。

## 8. 五种视图

### 8.1 列表视图

按事件逐行显示序号、时间、进程、PID、COM 口、方向、IRP/IOCTL、状态、长度、HEX、字符串和备注。支持列隐藏、排序、固定、调整宽度和按类型配色。

### 8.2 Line 视图

用可读文本逐项展示请求和完成信息。原始数据支持 2、8、10、16 进制，支持按指定字节数换行。

### 8.3 Dump 视图

显示偏移、字节区和文本区。可按端口、方向和指定时间间隔合并连续数据，也可选择显示非 Read/Write 事件。

### 8.4 终端视图

以指定编码显示 Read/Write 文本，分别配色。可显示时间、端口和方向，可自动换行、自动滚屏，并能冻结滚屏以便选择复制。

### 8.5 Modbus 视图

显示模式、站号、功能码、地址、数量、数据、校验结果、请求/响应/异常、COM 口和时间。支持标准功能码和保留为原始帧的自定义功能码。

RTU 不能假定一次 IRP 就是一帧。解析器按端口和方向分别缓存字节，综合波特率、3.5 字符时间、已知功能码长度和 CRC 确定帧边界。ASCII 使用冒号起始和 CRLF 结束，并检查 LRC。解析失败的字节必须保留为原始记录。

## 9. 复制体验

复制是核心功能，任何视图都不设置功能限制。

### 9.1 基本交互

- `Ctrl+C` 复制当前选择；单元格选择复制单元格，多行选择复制多行。
- 工具栏提供醒目的“复制数据”按钮和格式下拉菜单。
- 右键菜单同时提供常用复制格式。
- `Ctrl+Shift+C` 复制纯原始数据，避免多次打开菜单。
- 大量记录复制在后台格式化，显示进度，并允许取消。

### 9.2 复制格式

- 带空格 HEX，如 `01 03 00 00 00 02`。
- 连续 HEX，如 `010300000002`。
- 当前编码文本。
- 原始字节对应的 C 数组。
- Python `bytes` 字面量。
- 适合 Excel 的制表符分隔完整行。
- CSV 和 JSON。

多行复制可以选择附带时间、端口、方向、进程、序号和 IRP 名称。最后一次选择记为默认值。

### 9.3 数据可靠性

复制始终从原始事件模型读取，不从屏幕上被截断的文本反推。不可解码字节按所选策略显示转义符或替换字符，原始 HEX 保持精确。

## 10. 搜索和过滤

系统区分三层过滤：

1. 驱动捕获过滤：端口、IRP 主功能和 IOCTL 位图，用于减少内核事件量。
2. 会话记录过滤：决定哪些已捕获事件写入当前会话。
3. 视图显示过滤：不删除数据，只控制当前视图可见记录。

过滤条件支持端口、PID/进程、方向、IRP、IOCTL、状态、长度范围、HEX 字节模式和文本。Modbus 额外支持模式、站号、功能码、地址、请求/响应、异常和校验状态。

搜索不会改变过滤条件，可在当前视图、当前会话或原始数据中前后定位，并高亮命中内容。

## 11. 错误处理和恢复

- 驱动任何监控失败都不得改变原请求状态、长度、数据或完成顺序。
- 缓冲区溢出时记录累计和区间丢包数，界面以醒目状态提示。
- 服务断开后驱动继续使用有限缓冲；服务恢复时读取剩余事件并插入“服务中断”标记。
- 数据库写入失败时停止会话持久化并保留内存中的有限数据，不停止原串口通信。
- 设备拔出、重插或 COM 号变化时生成设备状态事件，允许会话重新绑定设备实例。
- 驱动安装失败必须回滚 Ports 类过滤器注册并恢复原注册表值。
- 卸载前检查服务、应用和驱动状态；无法安全移除时安排重启后删除。
- 应用异常退出后，下次打开可恢复最近的 WAL 会话并提示用户确认。

## 12. 性能和安全要求

- 驱动热路径不进行文件、注册表、网络或用户态同步操作。
- 所有内核缓冲区长度都来自可信边界检查，复制长度不得超过实际请求缓冲区。
- 驱动只保存监控所需的有限数据，单事件最大捕获长度可配置并明确标记截断。
- 服务批量读取、批量写库；界面每批刷新，避免为每个字节跨进程调用。
- UI 使用数据虚拟化，打开百万条记录的会话时不一次性创建全部行对象。
- 驱动设备和服务管道设置最小权限 ACL。
- 日志默认只保存在本机，不包含联网、账号或遥测功能。

## 13. 分阶段交付

### 阶段一：可用核心版

- KMDF 过滤驱动、测试证书和安装/卸载/诊断脚本。
- 后台服务和版本化驱动通信协议。
- 多串口选择与 Read、Write、关键配置 IOCTL 捕获。
- 阶段一的关键配置 IOCTL 至少覆盖波特率、线路控制、流控、超时、特殊字符、DTR/RTS 和清除缓冲区的查询或设置请求；其余标准串口 IOCTL 在阶段二补齐。
- 开始、暂停、继续、停止和清空。
- 列表、Dump、终端视图。
- HEX/文本搜索和完整复制功能。
- 最小会话保存与 CSV、TXT、原始二进制导出。
- 阶段一安装说明、快速开始和卸载恢复说明。

### 阶段二：功能对齐版

- 全部主要 IRP 和标准串口 IOCTL 解析。
- Line 视图和五视图联动基础设施。
- 完整会话保存、恢复、过滤、搜索、配色、栏目和格式设置。
- HTML、JSON 等全部导出格式和实时重定向。
- 大数据量虚拟化、统计、丢包提示和异常恢复。

### 阶段三：完整交付版

- Modbus RTU/ASCII 重组、解析、过滤和请求响应匹配。
- 免费安装包、卸载回滚和环境诊断工具。
- 实机兼容性、压力、长时间运行和恢复测试。
- 完整图文安装手册、使用手册、常见问题和开发构建说明。

阶段之间保持同一事件协议和会话模型，避免前一阶段成为一次性原型。

## 14. 测试策略

### 14.1 自动化测试

- Core 单元测试：编码、进制、HEX 搜索、过滤、复制格式、导出和 Modbus 校验。
- 会话测试：批量写入、异常退出、WAL 恢复、版本兼容和百万记录查询。
- 服务测试：事件协议版本、断线重连、批量读取、丢包标记和权限。
- 驱动测试：缓冲区边界、取消、失败状态、设备拔插、并发 Read/Write 和 IOCTL。

### 14.2 集成与实机测试

- 使用虚拟串口对生成可重复的双向数据。
- 使用至少一种常见 USB 转串口芯片和真实串口设备验证。
- 在原程序独占打开 COM 口时开始和停止监控，确认原程序无需重连。
- 对照发送端和接收端原始日志逐字节验证监控数据。
- 使用 Driver Verifier 检查内存、IRQL、I/O、PnP 和电源路径。
- 执行高吞吐、百万事件、服务重启、应用崩溃和设备热插拔测试。

## 15. 验收标准

阶段一完成需同时满足：

1. 在 Windows 10/11 x64 测试模式下能按文档完成安装、重启、诊断和卸载。
2. 被测程序独占打开串口时，监控程序能开始捕获且不要求改变 COM 口。
3. 测试数据的 TX/RX 字节与原始发送和接收日志一致；出现截断或丢包时必须显式标记。
4. 原通信在监控服务或桌面程序关闭、崩溃时仍继续工作。
5. 三种阶段一视图可稳定显示，选中记录可直接复制成所有规定格式。
6. 会话能保存、重新打开，并能导出 CSV、TXT 和原始二进制。
7. 安装、快速使用、卸载和常见安装错误均有可照做的中文说明。

最终版本完成需额外满足：

1. 公开手册列出的五种视图、会话、过滤、搜索、显示设置、导出、重定向和 Modbus 功能均已实现。
2. 所有视图均支持复制，复制结果来自原始数据且格式正确。
3. Driver Verifier 和约定的压力、恢复、热插拔测试通过。
4. 完整安装和使用手册与实际软件行为一致，并包含测试模式恢复步骤。
